## 3. 选型标准与决策矩阵

### 3.1 为什么选 Electron 而非 Tauri/CEF？

| 维度 | Electron | Tauri | CEF |
|------|----------|-------|-----|
| 生态成熟度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 开发效率 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 包体积 | ~150MB | ~5MB | ~80MB |
| 性能 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 原生能力 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 学习成本 | 低（JS全栈） | 中（Rust） | 高（C++） |

**决策**: 选 Electron — 项目核心是多用户浏览器管理，需要丰富的 Chromium 能力（webview、session 隔离），Electron 原生支持最好，团队 JS 全栈开发效率最高。

### 3.2 为什么用 electron-store 而非 SQLite？

| 维度 | electron-store | SQLite (better-sqlite3) |
|------|---------------|------------------------|
| 数据规模 | 适合中小规模（<100MB） | 适合大规模数据 |
| 查询能力 | 简单 key-value | 完整 SQL |
| 部署复杂度 | 零配置 | 需要 native 编译 |
| 数据模型 | JSON 灵活 schema | 结构化表 |

**决策**: 选 electron-store — 本项目数据量不大（用户配置、书签、历史、脚本），JSON 结构足够，零配置降低维护成本。若后续数据量增长，可平滑迁移到 SQLite。

## 4. Electron 架构设计

### 4.1 主进程职责

```
main.js
├── 应用生命周期管理
├── BrowserWindow 创建与管理
├── IPC Handler 注册
├── 模块初始化
│   ├── UserManager      → 用户 CRUD + 持久化
│   ├── SessionManager   → 会话创建/销毁/隔离
│   ├── FileManager      → 文件系统操作
│   ├── DownloadManager  → 下载任务管理
│   ├── BookmarkManager  → 书签 CRUD
│   ├── HistoryManager   → 浏览历史
│   ├── ScriptManager    → 脚本存储
│   ├── ActivityMonitor  → 用户活跃监控
│   └── ConfigManager    → 应用配置
└── 系统集成（托盘、通知、协议注册）
```

### 4.2 渲染进程职责

```
renderer/
├── App.jsx                 → 路由入口
├── layouts/
│   ├── MainLayout.jsx      → 侧边栏 + 内容区
│   └── BrowserLayout.jsx   → 浏览器标签页布局
├── pages/
│   ├── UserManager.jsx     → 用户管理页面
│   ├── Browser.jsx         → 浏览器页面（webview）
│   ├── FileManager.jsx     → 文件管理器页面
│   ├── DownloadManager.jsx → 下载管理器页面
│   ├── ScriptManager.jsx   → 脚本管理页面
│   └── Settings.jsx        → 设置页面
├── components/             → 可复用 UI 组件
├── store/                  → Redux store + slices
└── utils/                  → 工具函数
```

### 4.3 Preload 脚本（安全桥接）

```javascript
// preload.js — contextBridge 安全暴露 API
contextBridge.exposeInMainWorld('electronAPI', {
  // 用户管理
  getUsers: () => ipcRenderer.invoke('get-users'),
  saveUser: (user) => ipcRenderer.invoke('save-user', user),
  deleteUser: (id) => ipcRenderer.invoke('delete-user', id),
  
  // 会话管理
  createSession: (userId) => ipcRenderer.invoke('create-user-session', userId),
  activateUser: (userId) => ipcRenderer.invoke('activate-user', userId),
  
  // 文件管理
  getFiles: (path) => ipcRenderer.invoke('get-files', path),
  deleteFile: (path) => ipcRenderer.invoke('delete-file', path),
  
  // 下载管理
  addDownload: (url, savePath) => ipcRenderer.invoke('add-download', url, savePath),
  onDownloadProgress: (callback) => ipcRenderer.on('download-progress', callback),
  
  // 事件监听
  onNotification: (callback) => ipcRenderer.on('notification', callback),
});
```

## 6. 安全架构

### 6.1 会话隔离

```javascript
// 每个用户独立 session partition
const partition = `persist:user-${userId}`;
const session = session.fromPartition(partition);

// webview 使用独立 session
<webview partition={`persist:user-${userId}`} src={url} />
```

### 6.2 路径遍历防护

```javascript
// 文件操作前验证路径合法性
const path = require('path');
const ALLOWED_ROOT = os.homedir();

function validatePath(targetPath) {
  const resolved = path.resolve(targetPath);
  if (!resolved.startsWith(ALLOWED_ROOT)) {
    throw new Error('Path traversal detected');
  }
  return resolved;
}
```

### 6.3 Content Security Policy

```html
<meta http-equiv="Content-Security-Policy" 
  content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';">
```

### 6.4 Preload 安全暴露

- 使用 `contextBridge.exposeInMainWorld` 而非 `nodeIntegration: true`
- `contextIsolation: true`（默认）
- `sandbox: true`（推荐）

## 7. 版本控制策略

### 7.1 分支模型

```
main (生产) ← release/* ← develop (开发) ← feature/* / bugfix/*
```

| 分支 | 用途 | 合并方向 |
|------|------|----------|
| `main` | 生产代码，打 tag | ← release/* |
| `develop` | 集成分支 | ← feature/*, bugfix/* |
| `feature/*` | 功能开发 | → develop |
| `bugfix/*` | Bug 修复 | → develop |
| `release/*` | 发布准备 | → main + develop |

### 7.2 版本号规范（SemVer）

```
v主版本.次版本.补丁版本
v1.0.0  →  首个正式版本
v1.1.0  →  新增功能
v1.1.1  →  Bug 修复
```

### 7.3 提交规范

```
feat: 新增用户管理功能
fix: 修复下载暂停后无法恢复
docs: 更新 README
test: 添加会话管理单元测试
refactor: 重构文件管理模块
chore: 更新依赖版本
```

## 8. CI/CD 流水线

### 8.1 GitHub Actions 工作流

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  lint-and-test:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 18 }
      - run: npm ci
      - run: npm run lint
      - run: npm test -- --coverage
  
  build:
    needs: lint-and-test
    runs-on: windows-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: installer
          path: dist/*.exe
```

### 8.2 发布流程

```
1. develop 合并到 release/v1.x.x
2. CI 自动构建 + 测试
3. 构建通过 → 合并到 main
4. 打 tag (v1.x.x)
5. CI 自动构建安装包 + 上传 GitHub Releases
6. electron-updater 检测到新版本 → 用户端自动更新
```

## 10. 技术风险评估

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| Electron 版本兼容性 | 低 | 高 | 锁定 ^28.0.0，多版本 Windows 测试 |
| webview 会话隔离失败 | 中 | 高 | 充分测试 session.fromPartition，代码评审重点 |
| 大文件下载内存溢出 | 中 | 中 | 使用 stream 写入，限制并发数 |
| electron-store 数据损坏 | 低 | 中 | 写入前备份，异常恢复机制 |
| 第三方依赖漏洞 | 中 | 高 | npm audit 定期检查，锁定依赖版本 |
